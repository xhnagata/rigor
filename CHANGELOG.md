# Changelog

## 0.14.0 - 2026-07-13

- Implement ADR 0001 consumer enforcement ([#26](https://github.com/xhnagata/rigor/issues/26)): `scripts/verify-provenance.mjs` now reads the numeric repository id from `certificate.sourceRepositoryIdentifier` (the field real `gh 2.96.0` JSON actually carries; `sourceRepositoryOwnerIdentifier` is also available), keeping the pre-existing `sourceRepositoryID`/extension-OID keys as fallbacks so the hand-authored `positive.json` fixture keeps passing. It also pins and validates the `gh` CLI version through new exported pure functions `parseGhVersion`, `isSupportedGhVersion` (major `2`, minor `>= 90` by default, configurable via policy), and `validateGhShape` (rejects a non-array top level, an entry missing `verificationResult`/`statement`/`signature.certificate`, or a certificate with no recognized repository-id field), wired into `evaluateVerification` and the CLI's thin `main()`, which now fails closed when `gh` is absent, its version is unparseable/unsupported, or its JSON shape is unrecognized, and records the observed gh version on success. A new committed fixture, `test/fixtures/provenance/positive-real-v0.13.0.json`, is a real (not hand-authored) `gh attestation verify dist/rigor.cjs --format json` capture for the actual `v0.13.0` release (repository id `1296432215`, source/signer digest `e159a4baf82d19193ca6d473a5fc5edd92282796`, subject sha256 `8b3c17a3d968c6dc60ef717b1853d86023d1088a4d6a5011c72eba73df272675`), trimmed to only the fields the verifier reads (public certificate data, no secrets). New `scripts/install-verified.mjs` is a consumer-owned reference verified-install/managed-promotion wrapper — an enforcement boundary only when a consumer runs it, with its own held policy, from outside the plugin it protects — whose exported pure functions (`checkApproval`, `validateBreakGlass`, `checkSeedIntegrity`, `decideInstall`) decide `promote` (verification, approval, and seed integrity all pass), `keep-pinned` (the documented offline last-known-good exception: only an already-approved, fresh, previously verified pinned version, gated by policy), `break-glass` (only a valid, independently human-approved, unexpired, bounded-lifetime record, always recorded `verified: false`, gated by policy, and never reachable when verification merely-failed approval/seed-integrity on otherwise-verified bytes), or `refuse` (fail closed on everything else — a replayed/denied version, staleness beyond max age with no valid exception, a missing/invalid attestation, or a seed-integrity mismatch). Its thin `main()` reuses `scripts/verify-provenance.mjs`'s exported decision functions and `scripts/package-plugin.mjs`'s `collectPluginFiles`/`buildArchive` to re-package a promoted seed and confirm it matches the verified archive digest (closing the verify/execute TOCTOU gap) before printing or (`--launch`) executing `claude --plugin-dir <seed>` — experimentally confirmed to exist in Claude Code 2.1.207, the session-scoped explicit load that makes this boundary real, since the ordinary marketplace install/update flow was confirmed to expose no pre-activation verification hook over its cache. 58 offline tests in `test/provenance.test.ts` (all new tests plus every pre-existing test, none removed or weakened) cover the real-fixture positive/negatives, `validateGhShape`, `parseGhVersion`/`isSupportedGhVersion`, and every `install-verified.mjs` decision path with fail-closed negatives. `docs/provenance.md` gains a "Consumer enforcement (#26)" section (install/upgrade, migration, rollback by pinned commit/digest, break-glass, offline max age, and plainly-stated bypassability for unmanaged marketplace users); `docs/provenance.md` and `docs/release.md` now state that GitHub immutable releases are enabled by the owner (an out-of-band action; no `.github/**` change) while keeping the exact-source-commit/subject-digest pins and independent deny/staleness policy; `docs/threat-model.md` and `docs/mvp-limitations.md` update the #26/plugin-distribution status from blocked to implemented-with-precise-residual-limits; and `docs/adr/0001-provenance-trust-model.md` records the implementation status. No SLSA Build Level, native marketplace pre-activation verifier, or provider-attested runtime model identity is claimed; configured model identity stays `unverified`. `src/**`, `dist/**`, `bin/**`, `hooks/**`, `agents/**`, `skills/**`, `.claude-plugin/**`, and `.rigor/rigor-ci.cjs` are unchanged.

## 0.13.0 - 2026-07-13

- Implement ADR 0001 producer provenance ([#25](https://github.com/xhnagata/rigor/issues/25)): a dedicated tag-triggered `release.yml` signer generates keyless GitHub OIDC Artifact Attestations (SLSA v1 build provenance) over three subjects — the release bundle `dist/rigor.cjs`, a deterministic complete-plugin archive, and a detached release manifest. `scripts/package-plugin.mjs` is a pure-Node deterministic packager (a minimal ustar writer plus a normalized gzip header, no system `tar`) that emits `rigor-<version>.tar.gz` containing exactly the 21-file plugin set under one `rigor-<version>/` top directory with fixed modes, zeroed mtime, and bytewise-sorted entries, rejecting missing files, symlinks, non-regular files, and unexpected entries under the packaged roots; it also writes the detached `rigor-<version>.release-manifest.json` (validated against `schemas/release-manifest.v1.schema.json`), which is never self-referential (a manifest inside the archive could not record the archive's own digest). `scripts/verify-provenance.mjs` is a fail-closed consumer reference verifier whose pure decision logic (`evaluateVerification`, `checkCertificateExtensions`, `checkSubjectDigest`, `compareManifestToArtifacts`, `isDenied`, `freshnessOk`) enforces, from the returned certificate JSON, the numeric repository id (`1296432215`), the `push` trigger event, the tag pattern, and the github-hosted runner class — the checks the `gh` flags do not — and fails closed on a nonzero `gh` exit, empty output, unknown JSON shape, ambiguous/conflicting results, a denied digest, missing freshness, or any mismatch, never printing `verified` on failure. A separate owner-dispatched `provenance-smoke.yml` (protected `provenance-smoke` environment, `workflow_dispatch` only) attests only a clearly-marked synthetic non-production subject and builds no release artifact; production policy rejects its distinct signer/event/ref identity. Offline tests under `test/provenance.test.ts` and `test/fixtures/provenance/` cover deterministic packaging, the archive allowlist/modes, manifest schema positives/negatives, and every verifier negative (changed bundle, changed non-bundle plugin file, mismatched detached manifest, wrong source SHA, wrong signer, wrong predicate, absent attestation, self-hosted runner). Workflows use least privilege (`contents: read`, plus `id-token: write` and `attestations: write` on build jobs only; never `contents: write`) with every action pinned to a reviewed full commit SHA. No SLSA Build Level is claimed, immutable releases are documented but not enabled, runtime model identity stays unverified, and consumer enforcement ([#26](https://github.com/xhnagata/rigor/issues/26)) remains blocked, so this is producer provenance only, not an end-to-end distribution guarantee. New `docs/provenance.md` documents verification, consumer-held policy, root rotation, deletion/withdrawal, denylisting, offline staleness, and rollback ([#25](https://github.com/xhnagata/rigor/issues/25)).

## 0.11.0 - 2026-07-13

- Add a contamination-aware dogfooding evaluation benchmark and outcome-based routing calibration process. `rigor.evaluation-manifest.v1` defines a versioned, split-labelled task set (bounded categories, an initial 10-20 task selection with a documented expansion path to 30-100, per-task `calibration`/`holdout` split, optional isolated worktree/fixture references for selected cross-model comparisons, and required `owner`/`reviewInterval` fields). `rigor eval-report` deterministically joins each manifest task's recorded outcome, attempt, and routing-plan evidence and emits `rigor.evaluation-report.v1` with accepted changes as the denominator for every per-accepted-change rate, explicit missing-data counts (usage status, model identity, provider cost, elapsed, linkage, malformed artifacts, unknown relative-cost attempts — counted, never guessed), all four capability classes always comparable, and over-routing/under-routing/retry-cost/escaped-defect signals carrying their definitions and explicit numerators/denominators; every consumed evidence field is strictly validated and a wrong-typed or missing required field counts the artifact as malformed, evidence reads are realpath-contained under the selected evidence root, and holdout aggregates are marked evaluation-only. `rigor eval-replay` re-runs the pure routing selector under a proposed model-profiles file without invoking any model, diffing the complete selection (status, candidate, capability class, relative cost); it replays the calibration split by construction, reaches holdout only behind an explicit `--holdout-final` flag recorded in the output, never mixes splits, and fails closed when the selected split has no replayable plan. `rigor calibration-proposal` records an inert `rigor.calibration-proposal.v1` (fixed `status: "proposed"`, `approvalEffect: "none"`) that requires evidence, expected trade-offs, and rollback criteria, verifies each cited report/replay artifact hash against a supplied file and the selected manifest, and rejects holdout citations without an explicit final-evaluation marker; no code path applies a proposal to routing, thresholds, or profiles. `docs/evaluation.md` defines the task categories, contamination rules, the recurring calibration owner, and an every-release review interval, and a synthetic fixture tree under `test/fixtures/evaluation/` produces the sample report and replay entirely offline. Configured `relativeCost` remains an abstract comparison weight, never a price, token count, or verified provider charge. The implementation went through two recorded independent Codex review rounds (nine findings, then six residuals, all remediated and evidenced as `GH-16`/`GH-16-R2`/`GH-16-R3`) ([#16](https://github.com/xhnagata/rigor/issues/16)).

## 0.7.1 - 2026-07-12

## 0.7.0 - 2026-07-12

- Record deterministic failure fingerprints and cross-attempt progress so an unchanged implementation-failure loop is distinguished from attempts that resolve different failures, and regressions cannot be hidden by deleting or weakening tests. A new pure `src/fingerprint.ts` normalizes free-form check output (masking file paths, timestamps, UUIDs, hex runs, durations, and line/column noise, in a documented order), classifies each failure's category (implementation vs. infrastructure/timeout, never auto-`flaky`), derives an error class and bounded, noise-masked failed-test ids, and computes per-check and per-verification fingerprints from digests only. `rigor verify` now attaches these normalized failure facts and a `failureFingerprint` to the verification, persisting only digests and structured facts — never raw command output or secrets. `rigor attempt-finish` copies the deterministic facts from the linked verification, aggregates a failure category, and compares against the most recent prior attempt to record `progress` (`first`/`unchanged`/`reduced`/`expanded`/`incomparable`) plus weakening signals; `reduced` is fail-closed and only reported when parseable test counts confirm coverage did not shrink. Infrastructure and timeout failures are categorized separately and never counted as an unchanged implementation-failure loop, and the model-supplied `failureClass` stays separate and labeled as speculation. All new fields are additive, so existing v1 attempt and verification artifacts still parse ([#13](https://github.com/xhnagata/rigor/issues/13)).

## 0.6.0 - 2026-07-12

- Add the `/rigor:assess` Skill and feed validated assessments into `/rigor:orchestrate`. `/rigor:assess` gathers only the task statement and the repository files, tests, and schemas it names, then produces a `rigor.routing-input.v2` with `signals`, an explicit `confidence`, and 1..20 repository-path-anchored `evidence` entries, tracing every signal judgment to at least one observation. No CLI, schema, or deterministic-routing behavior changed; the routing gates added in [#10](https://github.com/xhnagata/rigor/issues/10) are reused as-is ([#11](https://github.com/xhnagata/rigor/issues/11)).

## 0.5.0 - 2026-07-12

## 0.4.0 - 2026-07-11

- Add a versioned `rigor.routing-input.v2` assessment input that makes model-generated task difficulty assessments reviewable and fail closed before routing selects a candidate. v2 replaces free-form `assessmentReasons` with an `assessment` object carrying an explicit `confidence` (`low`/`medium`/`high`) and 1..20 `evidence` items, each anchoring an `observation` to a repository-relative path (absolute and `..` paths are rejected). The `routing-decision` and `routing-plan` artifacts gain an assessment summary (optional on plans for pre-existing artifacts), and `docs/orchestration.md` documents the trust treatment of assessment evidence ([#10](https://github.com/xhnagata/rigor/issues/10)).
