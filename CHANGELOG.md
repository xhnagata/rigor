## 0.7.1 - 2026-07-12

- Add `docs/adr/0001-provenance-trust-model.md` (ADR 0001), the provenance trust model for Rigor's release and distribution surfaces. For release tag, source commit, `dist/rigor.cjs`, plugin distribution, local evidence, CI evidence, workflow identity, and Claude/Codex runtime model identity, it fixes a producer, an independent verifier, a trust root, and the protected identity, and states replay/freshness, revocation/rotation, offline, and fail-open/fail-closed policy per subject. It adopts GitHub Artifact Attestations carrying SLSA v1 build provenance for `dist/rigor.cjs` and a deterministically packaged complete plugin artifact (hooks, skills, agents, launcher, manifests), rejecting a checksum beside the artifact, a signed tag alone, an unauthenticated SLSA-shaped format, and blanket local-evidence signing as sufficient provenance. Implementation is split into independently testable follow-ups: [#25](https://github.com/xhnagata/rigor/issues/25) for producer provenance and [#26](https://github.com/xhnagata/rigor/issues/26) for consumer enforcement; the ordinary Claude Code marketplace install/update flow has no confirmed pre-activation verification point, which is documented as a blocker for #26 rather than hidden. Apart from a one-line `.codex` entry in `.gitignore`, this change is documentation only: no attestation, workflow, package, key, or SLSA Build Level claim is implemented or made, and configured model identity remains unverified; `docs/threat-model.md`, `docs/architecture.md`, `docs/release.md`, and `docs/mvp-limitations.md` now reference the ADR ([#24](https://github.com/xhnagata/rigor/issues/24)).

## 0.7.0 - 2026-07-12

- Record deterministic failure fingerprints and cross-attempt progress so an unchanged implementation-failure loop is distinguished from attempts that resolve different failures, and regressions cannot be hidden by deleting or weakening tests. A new pure `src/fingerprint.ts` normalizes free-form check output (masking file paths, timestamps, UUIDs, hex runs, durations, and line/column noise, in a documented order), classifies each failure's category (implementation vs. infrastructure/timeout, never auto-`flaky`), derives an error class and bounded, noise-masked failed-test ids, and computes per-check and per-verification fingerprints from digests only. `rigor verify` now attaches these normalized failure facts and a `failureFingerprint` to the verification, persisting only digests and structured facts — never raw command output or secrets. `rigor attempt-finish` copies the deterministic facts from the linked verification, aggregates a failure category, and compares against the most recent prior attempt to record `progress` (`first`/`unchanged`/`reduced`/`expanded`/`incomparable`) plus weakening signals; `reduced` is fail-closed and only reported when parseable test counts confirm coverage did not shrink. Infrastructure and timeout failures are categorized separately and never counted as an unchanged implementation-failure loop, and the model-supplied `failureClass` stays separate and labeled as speculation. All new fields are additive, so existing v1 attempt and verification artifacts still parse ([#13](https://github.com/xhnagata/rigor/issues/13)).

## 0.6.0 - 2026-07-12

- Add the `/rigor:assess` Skill and feed validated assessments into `/rigor:orchestrate`. `/rigor:assess` gathers only the task statement and the repository files, tests, and schemas it names, then produces a `rigor.routing-input.v2` with `signals`, an explicit `confidence`, and 1..20 repository-path-anchored `evidence` entries, tracing every signal judgment to at least one observation. It reports task characteristics only — it never names or selects a model, and evidence observations must state facts, never conclusions like "use model Z"; candidate selection remains exclusively a `rigor route` CLI decision. Before any `--record`, it validates the written input with `rigor route --dry-run` and stops (never loosening signals or confidence to force a pass) on malformed/unsupported/evidence-free/contradictory input (exit `3`) or on `requires-review`/`unroutable` (exit `2`), including low-confidence assessments, contradictory evidence, missing acceptance criteria, and required human decisions. `/rigor:orchestrate`'s routing-input argument becomes optional: when the caller supplies none, orchestrate obtains one via `/rigor:assess` and validates it via dry-run before `rigor route --record`, preserving every existing stop condition and the unverified configured-identity guarantee. No CLI, schema, or deterministic-routing behavior changed; the routing gates added in [#10](https://github.com/xhnagata/rigor/issues/10) are reused as-is ([#11](https://github.com/xhnagata/rigor/issues/11)).

## 0.5.0 - 2026-07-12

- Add `rigor availability`, a versioned `rigor.availability.v1` report that marks each configured candidate as exactly one of `available`, `unavailable`, `unknown`, or `incompatible`. Probing reads only documented, bounded local interfaces (a fixed set of environment variables: `CLAUDE_PLUGIN_ROOT`/`CLAUDE_CODE_ENTRYPOINT`, `CLAUDE_CODE_VERSION`, `ANTHROPIC_MODEL`, `RIGOR_CODEX_PLUGIN_PRESENT`, `RIGOR_CODEX_PLUGIN_VERSION`) and performs no installation, authentication, or network transmission; the `codex-plugin-cc` presence variables are an orchestrator-declared channel, not a direct plugin observation. Provider incompatibility is a static property of the provider type and is derived even when probing is unsupported. Tool/plugin version and observation time are recorded when observable and explicitly `null`/unknown otherwise, never fabricated; configured model identity is recorded as `unverified`. `rigor route --availability` filters `unavailable`/`incompatible` candidates with new `UNAVAILABLE`/`INCOMPATIBLE` reason codes (missing `codex-plugin-cc` excludes only candidates that require it) and records an `availabilityReportHash`; unsupported, failed, or format-changed probing derives `unknown` (never `available`) and never excludes a candidate, and an unroutable selection still stops with an explicit result instead of a silent substitution. Availability is an observation, not attestation: runtime model identity, reasoning effort, usage, and cost remain unverified/unknown. Schema changes are additive so existing v1 artifacts remain readable ([#12](https://github.com/xhnagata/rigor/issues/12)).

## 0.4.0 - 2026-07-11

- Add a versioned `rigor.routing-input.v2` assessment input that makes model-generated task difficulty assessments reviewable and fail closed before routing selects a candidate. v2 replaces free-form `assessmentReasons` with an `assessment` object carrying an explicit `confidence` (`low`/`medium`/`high`) and 1..20 `evidence` items, each anchoring an `observation` to a repository-relative path (absolute and `..` paths are rejected). Parsing fails closed for malformed, unsupported-schema, evidence-free, and contradictory assessments (high confidence cannot co-occur with critical ambiguity or weak verification). A low-confidence assessment yields an explicit `requires-review` routing outcome with no selection instead of silently selecting the economy class. Required capability still derives only from the assessment signals and stays independent of preflight risk; confidence is a reviewable judgment, not a calibrated probability, and evidence paths are validated as well-formed, not verified as true. Backward compatible: v1 inputs parse unchanged and route exactly as before, and routing plans recorded before this change parse with a legacy medium-confidence default. The `routing-decision` and `routing-plan` artifacts gain an assessment summary (optional on plans for pre-existing artifacts), and `docs/orchestration.md` documents the trust treatment of assessment evidence ([#10](https://github.com/xhnagata/rigor/issues/10)).

## 0.3.0 - 2026-07-11

- Add `rigor outcome`, an append-only outcome artifact that records the human-reported disposition of a task (accepted/rejected, accepted-without-model-code-changes, human-correction minutes, escalation count, review findings, revert and escaped-defect status) and links it to its attempt, verification, and review. It copies provider, model, capability class, and linked identifiers from the evidence rather than the input, derives `retryCount` from the linked attempt, and fails closed on inconsistent claims (accepted without a completed attempt or passing verification, reverted or escaped-defect while rejected, conflicting retry count, unavailable usage carrying measurements, inconsistent token totals). Tokens, provider cost, reasoning effort, and model identity are stored as measured-or-unavailable, configured model identity is `unverified`, and reported cost is a measurement rather than a Rigor-verified charge. Extend `rigor retrospect` to aggregate outcome artifacts into `outcomeTotals` and per-candidate `candidates` metrics with explicit rate numerators/denominators and data-completeness counts, tolerating malformed outcome files by counting them. Versioned `outcome-input.v1` and `outcome.v1` schemas accompany the command.
- Add `rigor release-check`, a deterministic pre-tag gate that verifies a clean worktree, synchronized `package.json` and `.claude-plugin/plugin.json` versions, a `CHANGELOG.md` entry for the requested version, a committed `dist/rigor.cjs` byte-identical to a fresh build, HEAD on the expected branch and optional pinned SHA, and a successful exact-SHA GitHub CI result via the read-only `githubReader`; the remote check is separable so unit tests exercise every finding with injected facts and no network, and the gate fails closed when CI is unverified. Document the protected-PR release path (release commits reach `main` only through a protected pull request with required checks, never a direct push) in [docs/release.md](docs/release.md) and both READMEs ([#8](https://github.com/xhnagata/rigor/issues/8)).

## 0.2.0 - 2026-07-11

- Fix the generated `.rigor/rigor-ci.cjs` entrypoint so direct GitHub workflow execution runs the verifier instead of exiting successfully without output. E2E now executes the generated bundle by its installed filename and requires parseable verifier output.
- Add recorded routing plans and `attempt-start`/`attempt-finish` around Claude or `codex-plugin-cc` implementation. Attempts enforce one active session, maximum attempts, duration and contract scope, mark configured execution identity unverified, and require linked passing verification before recording `completed`; `verify --dry-run` allows failed retries without consuming write-once verification evidence.
- Add policy-gated `consult-start`/`consult-finish` commands, append-only Codex consultation session/result artifacts, content-sensitive worktree hashing, and `/rigor:consult` plus `/rigor:orchestrate` Skills. Consultation finalization fails when content, changed paths, or HEAD differ, even when Git status still names the same modified path.
- Add advisory `rigor route --dry-run` with versioned routing-input, model-profile, routing-decision, attempt, and consultation schemas. The pure selector separates explicit complexity assessments from deterministic preflight controls, excludes candidates by purpose, capability, additional external transmission, and relative-cost budget, and invokes no model. Document the planned optional `codex-plugin-cc` boundary and its pre/post Git mutation checks.
- Add `rigor governance`, a read-only GitHub configuration verifier covering branch rulesets and classic protection, the required `rigor` status check, approval count, stale-approval dismissal, code-owner review, last-push approval, force-push/deletion blocking, sampled CODEOWNERS coverage of policy-protected globs (early warning, not proof), and deployment environment protection. GET-only against a fixed host with redirect refusal, a 10-second timeout, a response size limit, and a pagination guard that treats unfetched `rel="next"` pages as unverifiable; least-privilege token from the environment; fail-closed on unreadable configuration; the GitHub API is documented as an explicit trust boundary in the threat model ([#2](https://github.com/xhnagata/rigor/issues/2)).
