# MVP limitations and follow-up

Rigor 0.1.0 intentionally leaves these controls to follow-up work:

- Cryptographic runtime/model attestation, concurrent attempt coordination, and authoritative remote usage/cost collection. Empirical routing calibration now has a contamination-aware evaluation process ([evaluation.md](evaluation.md), [#16](https://github.com/xhnagata/rigor/issues/16)): `rigor eval-report` compares capability choices by accepted-change cost, `rigor eval-replay` shadow-replays proposed routing changes on the calibration split, and `rigor calibration-proposal` records inert, human-review-required proposals. It remains advisory: no code path applies a proposal, holdout tasks never fit thresholds, sample sizes are small and reported without statistical-confidence claims, and the CLI still invokes no model. Claude and `codex-plugin-cc` execution occurs only through explicitly invoked Skills; the CLI records configured identity as unverified and does not invoke a model or observe actual provider usage. `rigor availability` now probes candidate invocability through documented, bounded local interfaces and marks each candidate available/unavailable/unknown/incompatible, but this is an observation, not attestation: it does not prove which runtime model served a request, and runtime identity, reasoning effort, usage, and cost remain unverified/unknown.

- [Windows-native launcher and path/case testing](https://github.com/xhnagata/rigor/issues/1). The runtime logic is portable, but plugin execution is currently a POSIX shell script.
- [Cryptographic artifact signing, binary attestation, and SLSA provenance](https://github.com/xhnagata/rigor/issues/4). [ADR 0001](adr/0001-provenance-trust-model.md) adopts GitHub Artifact Attestations with SLSA v1 build provenance for `dist/rigor.cjs` and a complete distributable plugin package. Producer provenance is now **implemented** in [#25](https://github.com/xhnagata/rigor/issues/25): a tag-triggered signer workflow, deterministic packaging, a detached release manifest, and a fail-closed reference verifier with offline negative tests ([provenance.md](provenance.md)). No SLSA Build Level is claimed. Consumer enforcement ([#26](https://github.com/xhnagata/rigor/issues/26)) is still **outstanding**: the normal Claude Code marketplace path has no confirmed fail-closed point that verifies the exact cached bytes before hooks, skills, agents, or `bin/rigor` execute, so producer provenance does not create an end-to-end distribution guarantee until #26 implements and tests that consumer boundary. CI still independently re-derives Git facts; local evidence is not signed, and configured Claude/Codex identity remains distinct from unavailable provider-attested runtime identity.
- [Semantic detection of subtly weakened or low-quality tests](https://github.com/xhnagata/rigor/issues/3). MVP reliably flags deleted tests and mutation/removal of configured checks; semantic quality remains human review. [test-integrity.md](test-integrity.md) catalogs the wider surface. Five signals are collected in shadow mode, and #23 adds strict proposal, replay, activation, waiver, rollback, and same-diff tamper checks. The shipped active registry is absent and zero signals are promoted: four legacy non-firing events have no classifications, outcomes, or version manifest. There is no false-negative measurement without a labeled non-firing audit sample. The controls provide deterministic consistency under trusted CI and protected Git history, not human identity, empirical provenance, or administrator-resistant enforcement, so this limitation stands.
- [GitHub API verification of branch protection, rulesets, CODEOWNERS approval independence, and deployment environments](https://github.com/xhnagata/rigor/issues/2) is now available read-only via `rigor governance`. Remaining limits: classic branch protection is only readable with administration read scope (reported as unverifiable otherwise), CODEOWNERS matching is a documented last-match-wins subset whose coverage check samples one representative path per protected glob (an early warning, not proof of full glob coverage), and Rigor still makes no configuration writes.
- A managed remote metrics service. `retrospect` is local and redacted by design; teams must export only under an approved data policy.

None of these gaps is presented as an implemented guarantee.
