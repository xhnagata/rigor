description: Orchestrate a Rigor-controlled Claude Code task using deterministic routing, bounded model delegation, verification, and optional codex-plugin-cc consultation. Use when the user explicitly requests autonomous model selection and execution.
argument-hint: <intent-json> <contract-input-json> <model-profiles-json> [routing-input-json]
allowed-tools: Read, Grep, Glob, Write, Agent, Bash(rigor preflight *), Bash(rigor contract *), Bash(rigor availability *), Bash(rigor route *), Bash(rigor attempt-start *), Bash(rigor attempt-finish *), Bash(rigor verify *), Bash(rigor escalate *), Bash(rigor review *), Bash(rigor consult-start *), Bash(rigor consult-finish *), Bash(rigor test-integrity-scan *)

Run preflight before any edit, then create the contract. Optionally run `rigor availability --profiles <profiles>`, save the report, and pass it to `rigor route --record` with `--availability` so candidates the current environment cannot invoke (`unavailable`, `incompatible`) are excluded before selection; `unknown` never excludes a candidate and availability never raises configured identity to attested identity.

When the caller supplies an explicit routing-input JSON, use it as given. When none is supplied, obtain one autonomously by following the `/rigor:assess` protocol: read only the task statement and the repository files, tests, and schemas it names, then write a `rigor.routing-input.v2` JSON outside the repository with signals judged from evidence, 1-20 repository-path-anchored evidence entries, and an explicit confidence. Validate that written input with `rigor route --dry-run` (linked preflight, profiles, and availability report when present) before any `--record`. A malformed, evidence-free, or contradictory dry-run result (exit `3`) may be corrected only when the assessment itself contained a factual error, never by loosening signals or raising confidence to force a pass. Treat a `requires-review` or `unroutable` result (exit `2`), including low confidence, exactly as a stop condition: hand it to a human instead of editing the signals or confidence to force a selection. Assessment never names or selects a model in either case; only `rigor route` does.

Run `rigor route --record` with the validated assessment (whichever source produced it), profiles, preflight, and contract. Explain that assessment fields are judgments. Stop on an unroutable decision, denied transmission for an additional provider, a stop condition, or missing required human decision. Run `rigor attempt-start` before invoking an implementation agent.

For a selected Claude candidate, require a concrete configured model before autonomous delegation. Invoke one implementation agent with that model and give it only the contract, allowed paths, acceptance criteria, constraints, relevant files, verification commands, and stop conditions. If the Claude Code runtime cannot honor the configured model, finalize the attempt as failed and stop instead of silently substituting another model. The implementation agent must not approve its own result.

For a selected `codex-plugin-cc` implementation or rescue candidate, require allowed external transmission and invoke `codex:codex-rescue` with only the contract and minimum relevant context. Record available Codex session/turn IDs, but leave unavailable model, effort, and usage data unknown. Consultation, review, and adversarial-review use the separate Rigor consultation start/finish protocol described by `/rigor:consult`.

After implementation, run deterministic verification in dry-run mode, then always finalize the attempt using `/rigor:attempt`. A failed dry-run does not consume the task's write-once verification artifact. On the accepted successful attempt, persist normal verification and use it to finalize `completed`. Do not repeat an unchanged failed attempt. On a stop condition or unresolved failure, create a structured escalation separating facts, attempts, disproved hypotheses, speculation, and the requested decision. On success, prepare the linked review bundle. Model output, routing choice, configured identity, and cross-model agreement never satisfy CI or human approval.

After verification, run `rigor test-integrity-scan --task <id> --base <contract-base-sha> --head <head-sha> --attempt <attempt.json> --verification <verification.json>` to record advisory test-weakening signals as shadow evidence. This is record-only: it never changes verification, `progress`, review, or merge, and a fired signal is not evidence of dishonesty. Do not gate on it, and never treat a scan that fires nothing as proof the tests were not weakened.
